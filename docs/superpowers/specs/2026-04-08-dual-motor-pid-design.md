# 双电机 PID 配置与本地保存设计

## 背景

当前 PID 配置区只有一套公共输入框：

- 无论发送 `两轮同步`、`电机 A` 还是 `电机 B`，都复用同一组 `Kp / Ki / Kd` 输入框
- 参数查询后，`fill_params()` 只会把参数帧中的 `A_kp / A_ki / A_kd` 回填到这组输入框

这会带来两个实际问题：

1. 用户分别调好 A/B 两轮参数后，重新上电需要再次下发时，界面里通常只残留其中一侧的值
2. 软件重启后不会恢复上次输入过的 PID，用户需要重新输入

## 目标

- 在界面上同时呈现 `电机 A PID` 与 `电机 B PID` 两套输入框
- 支持 `两轮同步` 与 `独立配置` 两种模式
- `两轮同步` 模式下禁用 B 组输入，发送时直接使用 A 组参数同步下发
- 本地自动保存 A/B 两套 PID 与当前模式，下次启动自动恢复
- 参数查询继续显示设备当前参数，但不再强制覆盖本地待发送输入值

## 设计决策

### 1. PID 区布局

PID 配置区调整为三段结构：

- 顶部：模式选择
  - `两轮同步`
  - `独立配置`
- 中部：两列输入区
  - `电机 A`：`Kp / Ki / Kd`
  - `电机 B`：`Kp / Ki / Kd`
- 底部：发送按钮区

### 2. 模式行为

#### 两轮同步

- 仅允许编辑 A 组 PID
- B 组输入框禁用，但保留历史值显示
- 点击发送时使用 A 组参数构建 `CMD_SET_PID_BOTH`

#### 独立配置

- A/B 两组都可编辑
- 分别提供：
  - `设置 A`
  - `设置 B`
- 分别构建 `CMD_SET_PID_A` 与 `CMD_SET_PID_B`

### 3. 本地保存

采用自动保存与自动恢复，不增加额外保存按钮。

保存内容：

- `pid_mode`
- `motor_a.kp / ki / kd`
- `motor_b.kp / ki / kd`

保存时机：

- 任意 PID 输入值变化时
- 模式切换时

恢复时机：

- `CommandPanel` 初始化时读取本地配置
- 如果不存在配置文件，则使用默认值

### 4. 配置文件位置

本地配置文件放在项目目录下：

- `settings/pid_config.json`

设计理由：

- 文件结构简单，可读可调试
- 不引入额外依赖
- 与项目工作目录一起保留，迁移方便

### 5. 参数查询与回填

`固件参数面板` 继续作为“设备当前值”的展示来源：

- 查询参数后继续更新 `ParamPanel`
- `ParamPanel` 仍显示设备当前 A/B PID 与其他参数

`PID 输入框` 改为“本地待发送值”：

- 不再因为收到参数帧就自动覆盖
- 避免本地保存或手动准备的 A/B 两套值被意外冲掉

为保留从设备当前值载回输入区的能力，在 PID 配置区增加两个显式动作：

- `载入设备 A`
- `载入设备 B`

行为：

- `载入设备 A`：把最近一次参数帧中的 `A_kp / A_ki / A_kd` 写入 A 组输入框
- `载入设备 B`：把最近一次参数帧中的 `B_kp / B_ki / B_kd` 写入 B 组输入框

## 数据语义

- PID 输入框：用户准备下发的本地值
- 固件参数面板：设备当前实际值

两者职责分离后，用户能清楚区分“准备发什么”和“设备现在是什么”。

## 验收标准

- PID 面板同时显示 A/B 两套输入框
- `两轮同步` 模式下 B 输入框禁用，但不清空
- `独立配置` 模式下 A/B 可分别发送
- 软件重启后恢复上次 A/B PID 与模式
- 查询参数不会自动覆盖输入框
- 用户可显式把设备 A/B 当前 PID 载入对应输入框
